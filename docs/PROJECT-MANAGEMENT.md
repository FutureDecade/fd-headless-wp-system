# 项目管理边界

## 结论

`/Users/admin/Projects/FutureDecade/fd-headless-wp-system` 应该作为独立的交付仓库使用。

不要把它做成一个“把所有源码复制一份进来”的总目录。

## 为什么不能把所有项目都塞进来

如果把前端、主题、插件、websocket 都复制一份进这个目录，后面会出现 4 个问题：

1. 代码会重复
2. 版本边界会混乱
3. 修一个问题不知道该改哪个目录
4. 交付仓库会退化成新的大杂烩

## 推荐的管理模型

### 1. 业务代码仓库

这些仓库各自独立演进：

- `fd-frontend`
- `fd-websocket`
- `fd-theme`
- `fd-member`
- `fd-payment`
- `fd-commerce`
- 其他自研插件

### 2. 交付仓库

这个仓库只负责：

- 定义系统怎么被部署
- 定义系统需要哪些环境变量
- 定义服务之间怎么连接
- 定义安装和升级流程

### 3. 资产进入交付仓库的方式

短期：

- 通过本地相对路径或构建产物复用现有代码

中期：

- 前端用版本化镜像
- websocket 用版本化镜像
- 主题和插件用 zip 包或自定义 WordPress 镜像

长期：

- 所有自研插件/主题都有自己的 Git 来源和版本号
- 交付仓库只引用版本，不直接承载主开发工作

## 现阶段最小化改动原则

现阶段最稳的做法是：

1. 不改现网目录结构
2. 不合并现有仓库
3. 不先迁移生产
4. 先在交付仓库内完成统一编排

## 你应该怎么理解这个仓库

它更像：

- system repo
- deployment repo
- orchestration repo

而不是：

- monorepo
- all-in-one source repo
