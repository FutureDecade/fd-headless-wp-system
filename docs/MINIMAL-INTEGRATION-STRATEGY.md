# 最小改动整合策略

这份文档回答一个核心问题：

- 如何在不破坏现有生产运行的前提下，逐步把系统整理成可交付的一键部署方案

## 最终结论

不要做这两件事：

- 不要把 `fd-frontend` 改造成整套系统的总仓
- 不要把 `/opt/wordpress` 运行目录直接演变成交付仓库

应该做的事是：

- 保持业务仓库继续独立
- 用 `fd-headless-wp-system` 作为新的顶层交付层
- 先统一编排，再统一资产，最后才做正式迁移

## 为什么这样最稳

因为当前系统的风险不是“单个组件不会跑”，而是：

1. 组件分散
2. 域名和配置写死
3. WordPress 资产主要还在服务器里
4. 生产环境已经在线，不能冒险大改

所以最稳妥的顺序必须是：

1. 固定系统边界
2. 固定交付结构
3. 提取缺失资产
4. 在新环境验证
5. 最后才迁移生产

## 现阶段不能做的错误动作

### 1. 不能先改生产目录结构

不能在当前线上服务器上直接把：

- `/opt/fd-frontend`
- `/opt/wordpress`

合并、挪动、重命名。

原因：

- 这会直接影响现网运行路径
- 会增加回滚复杂度

### 2. 不能先复制一堆源码进交付仓库

`fd-headless-wp-system` 不是新的 monorepo。

如果把前端、主题、插件都复制一份进来，后面一定会产生：

- 双份代码
- 版本不一致
- 修改来源不清

### 3. 不能先做生产一键安装

现在还没完成：

- WordPress 主题/插件版本标签治理
- WordPress 主题/插件接入交付流程
- ACF 配置导出治理

所以现在写“最终 install.sh”还太早。

## 当前推荐架构

### 业务层仓库

- `fd-frontend`
- `fd-websocket`
- `fd-theme`
- `fd-member`
- `fd-payment`
- `fd-commerce`

### 交付层仓库

- `fd-headless-wp-system`

交付层负责：

- 顶层 `docker-compose.yml`
- `.env.example`
- Nginx 模板
- 安装脚本
- 环境检查
- 交付文档

## 分阶段整合路线

## Phase A：交付层成型

目标：

- 固定交付仓库边界
- 固定系统级 compose
- 固定环境变量规范
- 固定 CI 校验

这个阶段已经开始，并且 CI 已经通过。

成功标准：

- 仓库独立
- 有顶层 compose
- 有 `.env.example`
- 有基础脚本
- 有 GitHub Actions

## Phase B：组件来源固化

目标：

- 明确每个组件未来从哪里进入交付系统

推荐做法：

- 前端：继续沿用已发布镜像
- WebSocket：直接引用已发布 GHCR 镜像
- WordPress：先保留官方镜像，主题/插件先走 zip 制品，再决定是否构建自定义 WordPress 镜像

成功标准：

- 每个组件有明确来源
- 不再依赖“线上目录手工复制”

## Phase C：WordPress 资产产品化

目标：

- 把真正决定系统可复原的 WordPress 资产提取出来

包括：

- `fd-theme`
- `fd-*` 自研插件
- ACF 字段组
- Gravity Forms 配置
- 必要演示数据策略

成功标准：

- 新环境不依赖线上 `wp-content` 临时拷贝

## Phase D：Staging 验证

目标：

- 在全新测试机上验证统一编排方案

这里要验证：

- 容器能否一起启动
- 反向代理是否正常
- WordPress / Frontend / WebSocket 是否连通
- 数据初始化是否能跑通

成功标准：

- 从零搭建一次可复现

## Phase E：安装器与迁移

目标：

- 在经过 staging 验证后，再补完整安装器和生产迁移方案

成功标准：

- 有清晰的正式迁移窗口
- 有备份和回滚方案

## 当前最推荐的具体策略

### 前端

保持当前模式：

- 继续作为独立业务仓库
- 继续有自己的 CI/CD
- 交付层只引用其镜像

这是当前最成熟、风险最低的部分，不应该先动。

### WebSocket

这是最应该尽快治理的部分。

原因：

- 它已是独立服务
- 体量小
- 但当前没有 Git 来源

正确动作：

- 继续保持独立仓库
- 由交付层引用其镜像

### WordPress

WordPress 不能直接“迁到前端里”，也不能直接拿当前运行目录当交付模板。

正确动作：

- 先保留官方镜像思路
- 继续单独治理 `fd-theme` 和 `fd-*` 插件
- 先把主题/插件固定成 zip 制品来源
- 把 WordPress 从“运行现场”变成“可交付资产”

## 当前阶段的成功标准

你现在不应该问：

- 能不能今天就一键部署生产

你现在应该问：

- 系统边界是不是已经固定
- 组件来源是不是越来越清楚
- 每一步是不是都能不碰生产推进

只要这三个答案持续是“是”，路线就是对的。
